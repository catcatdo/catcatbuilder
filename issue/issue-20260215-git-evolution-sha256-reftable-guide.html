20년 된 Git, 왜 지금 다시 배워야 할까: SHA-256과 Reftable 전환 체크리스트

릴황 테크 이슈 · 2026년 2월 15일 · 버전관리 운영 가이드

Git은 당장 명령어가 바뀌는 도구라기보다 저장소 내부 구조가 서서히 바뀌는 인프라에 가깝습니다. 그래서 핵심은 "언제 업그레이드하느냐"보다 "무엇부터 호환성 점검하느냐"입니다.

1. 지금 Git 변화가 중요한 이유

팀의 저장소 규모가 커질수록 성능 문제는 코드 품질이 아니라 운영 구조에서 생깁니다. 브랜치·태그가 급증하고, CI 잡이 늘고, 바이너리 아티팩트가 섞이기 시작하면 기존 기본값만으로는 속도와 안정성을 동시에 유지하기 어렵습니다.

최근 논의되는 변화의 핵심은 세 가지입니다. 해시 체계를 강화해 무결성을 높이고, 레퍼런스 저장 구조를 개선해 대형 저장소의 병목을 줄이며, 대용량 객체 처리 전략을 분리해 운영비와 복구 시간을 제어하는 것입니다.

2. SHA-256 전환: 먼저 확인할 5가지

보안 전환은 저장소만 바꾼다고 끝나지 않습니다. CI/CD, 코드 스캔, 릴리스 스크립트, 내부 툴링이 모두 커밋/오브젝트 ID 형식에 의존할 수 있기 때문입니다.

점검 항목 왜 중요한가 실무 기준
호스팅 플랫폼 지원 여부 원격 저장소가 미지원이면 전환 테스트가 중단됨 플랫폼 릴리스 노트 기준으로 지원 범위를 먼저 확정
CI 스크립트의 해시 길이 가정 고정 길이 파싱 로직이 배포 파이프라인을 깨뜨릴 수 있음 정규식/파서가 알고리즘 종속인지 코드 검색으로 확인
웹훅·봇 연동 커밋 ID 전달 포맷이 바뀌면 알림/자동화 누락 가능 샌드박스 저장소에서 이벤트 왕복 테스트 수행
백업/복구 정책 복구 시점에 해시 체계 혼재가 있으면 복원 절차가 복잡해짐 백업 메타데이터에 object format 정보를 명시
팀 문서와 리뷰 가이드 운영 지식이 코드보다 늦게 갱신되면 실수가 반복됨 운영 문서에 전환 상태/제약/롤백 절차를 고정
git rev-parse --show-object-format git fsck --full git count-objects -vH

3. Reftable은 어떤 팀에 유리한가

Reftable은 refs 파일 구조를 재정리해 대규모 브랜치 환경에서 비용을 줄이는 접근입니다. 모든 팀에 즉시 필요하진 않지만, 아래 조건이 많을수록 체감 효과가 큽니다.

브랜치 수가 많은 조직

릴리스/핫픽스/실험 브랜치가 동시 운영되며 refs 관리 작업이 빈번한 팀.

모노레포 + 대형 CI

하루 수백~수천 파이프라인이 돌고 브랜치 생성·삭제가 자동화된 환경.

서버 저장소 운영 비중이 높은 팀

개발자 로컬보다 중앙 저장소 운영 안정성이 SLA에 직접 연결되는 구조.

성능 개선은 저장소 내부 구조만으로 끝나지 않습니다. 브랜치 수명 정책(자동 삭제, 보호 브랜치 최소화, 릴리스 태깅 기준)을 함께 정리해야 효과가 유지됩니다.

4. 대용량 파일은 Git에 "다" 넣지 않는다

영상, 모델 파일, 데이터 덤프 같은 대용량 바이너리는 텍스트 중심 버전관리 철학과 충돌하기 쉽습니다. 핵심은 코드 이력과 아티팩트 보관을 분리하는 운영 방식입니다.

  • 코드/메타데이터: Git 본체에 유지
  • 대형 바이너리: LFS 또는 오브젝트 스토리지로 분리
  • 클론 전략: 부분 클론/얕은 클론으로 온보딩 시간 단축
  • 복구 전략: 코드 복구와 바이너리 복구 경로를 별도로 문서화

이렇게 분리하면 저장소 크기 폭증, 긴 클론 시간, 네트워크 비용 급등을 동시에 완화할 수 있습니다.

5. 30-60-90일 전환 플랜

  1. 30일: 현재 Git 사용 실태 진단 (훅, CI, 릴리스 스크립트, 대용량 파일 현황)
  2. 60일: 샌드박스 저장소에서 해시/refs/클론 전략 A/B 테스트
  3. 90일: 팀 문서 업데이트, 점진 배포, 롤백 기준 확정 후 단계적 전환

포인트는 "빅뱅 교체"가 아니라 "업무 중단 없이 부분 전환"입니다. 전환 성공률은 기술 난이도보다 실행 순서에서 갈립니다.

결론

Git 변화의 본질은 새로운 명령어가 아니라 운영 기준의 업그레이드입니다. SHA-256, Reftable, 대용량 객체 전략을 각각 따로 보지 말고, 저장소 신뢰성·성능·비용의 균형 관점에서 한 번에 설계해야 합니다.

팀이 지금 해야 할 일은 간단합니다. 호환성 체크리스트를 만들고, 작은 저장소에서 먼저 검증하고, 운영 문서와 자동화를 같은 속도로 업데이트하는 것입니다.
